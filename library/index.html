A Chatbot Answers. An Agent Acts.

When you talk to ChatGPT or Claude in a browser, you type a message, get a response, and repeat. The model has no ability to go do things on your behalf. It can only generate text.

An agent is different because it has tools. It can read files, run shell commands, search the web, create pull requests, send Slack messages, query databases. It looks at the result of those actions and decides what to do next. That loop of thinking, acting, and observing is what makes something an agent rather than a chatbot.

The Agent Loop

Every agent, regardless of which framework built it, runs the same basic cycle:

This is called the ReAct pattern (Reasoning + Acting). The model reasons about what it needs to do, picks a tool, executes it, looks at the result, and reasons again. Each cycle is called a "turn."

The number of turns matters. A chatbot always takes exactly one turn. An agent might take 3 turns to answer a question (search, read, summarize) or 500 turns to implement an entire feature (read code, write code, run tests, fix errors, commit, push).

Tools: The Hands of an Agent

A tool is a function the agent can call. Some examples:

  • Read – read a file from disk
  • Edit – modify a file with a precise diff
  • Bash – run a shell command
  • Grep – search for patterns in code
  • WebSearch – search the internet
  • WebFetch – load and parse a web page

The agent sees a list of available tools and their descriptions. When it decides to use one, it formats the input as JSON, the tool executes, and the output goes back into the conversation. The model never runs code directly. The tool execution happens in a separate runtime that the agent framework manages. Tools are what separate an agent from a chatbot. Without tools, a model can only generate text. With tools, it can change the world.

Permissions and Safety

Because agents can do real things (delete files, push code, send messages), they need guardrails. Most agent frameworks have a permission system that controls which tools the agent can use and under what conditions.

Some tools are "safe" and get auto-approved, like reading files. Others are risky, like running arbitrary shell commands, and require explicit approval. You can configure this per-agent, per-tool, or per-context.

Context and Memory

Agents have a context window, the same as any LLM conversation. Every tool call, every result, every piece of reasoning takes up space in that window. Long-running agents can fill it up, which is why most frameworks support "compaction," where older parts of the conversation get summarized to free up space.

Some agents also have persistent memory that survives across conversations. This might be a file on disk, a database, or a structured memory system.

Multi-Agent Systems

Sometimes one agent hands off work to another. A "coordinator" agent decides what needs to happen, then spawns specialized agents to do parts of the work.

For example, a code review agent might spawn a "security reviewer" subagent and a "performance reviewer" subagent in parallel. Each one has its own context, its own tools, and its own focus. The coordinator collects their reports and synthesizes a final review.

Common Frameworks

  • Claude Agent SDK – Anthropic's framework for agents powered by Claude. Built-in tools, session management, permission controls.
  • LangChain / LangGraph – Popular open-source framework supporting multiple LLMs with a large integration ecosystem.
  • OpenAI Agents SDK – OpenAI's framework for GPT-powered agents with tool use and handoffs.
  • CrewAI – Focused on multi-agent workflows with role-based agents.
  • Custom wrappers – Many teams build thin wrappers around an LLM's CLI or API, skipping the framework entirely.

Takeaways

  • An agent is an LLM that can use tools and loop until a task is done
  • The ReAct pattern (think, act, observe, repeat) is the core of every agent
  • Tools are functions the agent can call: file I/O, shell commands, APIs, web searches
  • Permission systems control what the agent is allowed to do
  • Subagents let you split complex tasks across specialized workers
  • You can build agents with a framework or by wrapping a CLI/API directly

Three Layers

There are three main pieces to understand: Claude (the model), Claude Code (the interactive tool), and the Claude Agent SDK (the library for building your own agents). On top of all that sits MCP, a protocol that lets agents connect to external tools and data sources.

Claude: The Model

Claude is the large language model that Anthropic builds. The current generation includes three sizes: Opus (the most capable), Sonnet (balanced), and Haiku (fast and lightweight). When any Claude-based agent runs, one of these models is doing the reasoning.

You access Claude through the Anthropic API. You send it messages, it generates responses. If you give it tool definitions, it can also generate "tool use" blocks, structured requests to call a specific tool with specific arguments. Your code executes the tool and sends the result back. That back-and-forth is the agent loop.

Claude Code: The Product

Claude Code is Anthropic's interactive coding agent. You can use it as a CLI in your terminal (claude command), as a VS Code or JetBrains extension, as a desktop app, or through the web.

It has a built-in set of tools (Read, Edit, Write, Bash, Glob, Grep, WebSearch, WebFetch, and others) and a permission system that controls what it's allowed to do without asking.

Headless Mode

Claude Code supports a "headless" mode where you pass a prompt via the -p flag and get structured output:

claude -p "Fix the login bug in auth.ts" \
    --output-format stream-json \
    --max-turns 50

The Claude Agent SDK

The Agent SDK is a Python and TypeScript library that gives you programmatic access to the same agent loop that powers Claude Code. You import it, configure tools and permissions, pass a prompt, and get back a stream of messages as Claude works through the task.

from claude_agent_sdk import query, ClaudeAgentOptions

async for message in query(
    prompt="Find all TODO comments and create issues",
    options=ClaudeAgentOptions(
        allowed_tools=["Read", "Glob", "Grep", "Bash"],
        max_turns=50
    )
):
    print(message)

The SDK handles tool execution automatically. You tell it which tools are available, Claude decides when to use them, and the SDK runs the tool and feeds the result back.

Sessions

Every agent conversation is persisted as a "session" on disk (JSONL files). You can resume a session later with full context, meaning the agent picks up exactly where it left off.

Subagents

The SDK supports spawning subagents for subtasks with their own system prompts, tool sets, and model preferences. Each subagent runs in its own context and can run in parallel.

Hooks

Hooks let you intercept events during the agent loop. You can run code before a tool executes (to block dangerous operations), after a tool returns (to log results), or at various other points. Common uses: audit logging, blocking writes to production paths, injecting extra context, or forwarding notifications to Slack.

Model Context Protocol (MCP)

MCP is an open standard for connecting agents to external tools and data. It works like a plugin system. An MCP server exposes tools (like "list GitHub issues" or "query the database"), and any MCP-compatible agent can discover and call those tools. Think of MCP the same way you think of USB for hardware: a universal interface that lets any agent talk to any tool, as long as both sides speak the protocol.

  • Stdio – local processes, communication over stdin/stdout
  • HTTP/SSE – remote servers with server-sent events
  • In-process – tools defined directly in your code

The Permission Model

Safety is layered. The SDK evaluates permissions in this order:

  1. Hooks – can block or modify any tool call before it runs
  2. Deny rules – always block, regardless of other settings
  3. Permission mode – global behavior (prompt, auto-deny, auto-approve)
  4. Allow rules – pre-approve specific tools

SDK vs. CLI

The CLI is great for interactive use, one-off scripts, and simple automations. The SDK is better when you need fine-grained control, custom tool definitions, hooks, or tight integration with an existing application. Many teams start with the CLI and move to the SDK as their needs grow.

Takeaways

  • Claude is the model. Claude Code is the product. The Agent SDK is the library.
  • Claude Code can run headless with -p and --output-format stream-json
  • The Agent SDK manages the tool loop, sessions, permissions, and hooks for you
  • MCP is a plugin protocol that lets agents connect to external tools and data
  • Permissions are layered: hooks, deny rules, permission mode, allow rules

A Brief History of AI Agents

The idea of autonomous software agents is older than most people realize. The current wave of LLM-powered agents draws from nearly 70 years of research. Here's how we got here.

Symbolic AI and Planners (1958 - 1980s)

The earliest agent-like systems were symbolic. They reasoned about the world using logical rules and planned sequences of actions to reach goals.

1958 saw John McCarthy propose the Advice Taker, one of the first descriptions of a program that could take in knowledge and decide what to do with it. It never became a real system, but it outlined what an agent should be: something that receives information, reasons about it, and acts.

STRIPS (1971, Stanford Research Institute) was the first practical automated planner. You described a goal and a set of possible actions with preconditions and effects, and STRIPS would figure out a sequence of steps to reach that goal. Its action representation became the foundation for planning research for decades.

SHRDLU (1972, Terry Winograd at MIT) could understand natural language commands and manipulate blocks in a simulated world. It was impressive in its narrow domain but completely brittle outside of it. This brittleness became the defining limitation of the entire symbolic era.

Through the late 1970s and 1980s, expert systems like MYCIN (medical diagnosis, 1976) and R1/XCON (computer hardware configuration, 1980) showed that rule-based agents could be commercially useful. But every rule had to be hand-coded. Scaling was painful, and the systems broke in unexpected ways when they encountered situations their authors hadn't anticipated.

Reactive Agents and BDI (1986 - 1995)

The 1980s brought a backlash against the symbolic approach.

In 1986, Rodney Brooks at MIT published his work on subsumption architecture. His argument: forget internal world models. Build agents as layered sets of simple reactive behaviors, where each layer handles a specific concern (avoid obstacles, wander, explore). Complex behavior emerges from the interaction of simple rules, without any central planner. He proved this worked on real robots.

At the same time, the BDI model (Belief-Desire-Intention) emerged from philosophy (Michael Bratman, 1987) and was formalized computationally by Rao and Georgeff in 1991. BDI agents maintain beliefs about the world, desires they want to achieve, and intentions they've committed to. This framework is still used in multi-agent system design today.

In 1995, Wooldridge and Jennings published "Intelligent Agents: Theory and Practice," a survey that defined the vocabulary the field still uses: autonomy, reactivity, pro-activeness, social ability. If you've heard anyone describe what makes an "agent" an agent, the definitions probably trace back to this paper.

Multi-Agent Systems (1996 - 2005)

With individual agents better understood, researchers turned to making them work together.

FIPA (Foundation for Intelligent Physical Agents) standardized how agents communicate, defining an Agent Communication Language with message types borrowed from speech act theory: request, inform, propose, accept, reject. JADE (2000), a Java framework implementing FIPA standards, became the dominant platform for multi-agent research in academia.

This era produced foundational work in coordination: auction mechanisms for resource allocation, contract nets for task distribution, cooperative problem-solving protocols. Most of these ideas sat in academic papers for 20 years before LLM-based multi-agent frameworks rediscovered them.

Reinforcement Learning (2013 - 2019)

Deep learning changed everything. Instead of hand-coding agent behavior, you could let agents learn from experience.

DeepMind's DQN (2013, published in Nature 2015) played Atari games at superhuman level by learning directly from raw pixels. No rules, no heuristics, just a neural network learning which actions lead to higher scores.

AlphaGo (2016) defeated Lee Sedol 4-1 in Go, a game that had been considered too complex for brute-force search. It combined Monte Carlo Tree Search with deep neural networks trained through self-play. AlphaZero (2017) generalized the approach to chess, shogi, and Go, learning entirely from self-play with zero human game knowledge.

OpenAI Five (2019) beat world champions at Dota 2. DeepMind's AlphaStar reached Grandmaster level in StarCraft II the same year. These proved that learned agents could handle real-time, incomplete-information, multi-player environments at the highest level.

What these systems had in common: they were narrow. Each one was trained for a single game or environment. General-purpose agents that could handle arbitrary tasks were still out of reach.

LLM-Based Agents (2022 - Present)

Large language models changed what "general-purpose" meant for agents.

The pivotal paper was ReAct (Yao et al., October 2022), which showed that LLMs could interleave chain-of-thought reasoning with tool-use actions in a loop. The model reasons about what to do, calls a tool, reads the result, and reasons again. This Reason+Act pattern became the backbone of every modern agent framework.

Toolformer (Schick et al., Meta, February 2023) demonstrated that LLMs could learn when and how to call APIs on their own, without being told at each step.

Auto-GPT (March 2023) was the first agent that captured mainstream attention. It hit 100,000+ GitHub stars within weeks by attempting fully autonomous task execution with GPT-4. The practical reliability was limited, but it proved that the concept had broad appeal and got a lot of developers thinking about what was possible.

June 2023 brought OpenAI's function calling API, which gave LLMs a structured, reliable mechanism for invoking tools. This replaced the fragile prompt-based parsing that earlier agents relied on.

Through 2024 and 2025, the ecosystem matured. Anthropic released Claude Code as a full coding agent. Frameworks like LangChain, CrewAI, and AutoGen made multi-agent orchestration accessible. Microsoft's Magnetic-One and Anthropic's computer use capabilities pushed agents toward general-purpose computer interaction.

The current state (2025-2026): agent architectures combine ReAct-style loops, tool use, persistent memory, and multi-agent coordination. The big open problems are reliability (agents still make mistakes on multi-step tasks), evaluation (how do you benchmark something this open-ended), and safety (how much real-world authority should you delegate to a model).

What Connects All of This

Modern LLM agents are not a clean break from the past. They embed planning (from STRIPS), goal-directed behavior (from BDI), multi-agent coordination (from the FIPA era), learned policies (from RL), and natural language interfaces, all unified through the language model as a general-purpose reasoning engine.

The difference is that previous generations of agents were each built for one task or domain. LLM agents get general-purpose reasoning for free from the pre-trained model, and they get domain-specific capability through tools. That combination is why they feel like a qualitative shift, even though every individual piece has been around for years.

Timeline

  • 1958: Advice Taker proposed. 1971: STRIPS. 1972: SHRDLU.
  • 1986: Brooks' reactive agents. 1991: BDI formalized. 1995: Wooldridge & Jennings survey.
  • 2000: JADE framework. FIPA standards for agent communication.
  • 2013-2019: DQN, AlphaGo, AlphaZero, OpenAI Five, AlphaStar.
  • 2022: ReAct paper. 2023: Auto-GPT, Toolformer, function calling.
  • 2024-2026: Claude Code, mature frameworks, multi-agent orchestration.
